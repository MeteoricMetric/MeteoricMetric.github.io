# ADR-0001: Adopt Astro + TypeScript + vanilla-CSS-with-tokens stack

## Status
Accepted — 2026-04-30 (Shane)

## Context

The merricstrough.com v1 site is a single 89-line `index.html` with all-inline CSS, deployed to GitHub Pages. CLAUDE.md v1.0 §3.2 prescribed "no framework yet" — a deliberate choice when the site was 5 minutes of setup and Merric was just starting to learn web fundamentals.

Two things have changed since:

1. **Scope.** The v2 redesign targets Awwwards-SOTD / SpaceX / NASA Genesis Mission / Cerebras / OpenAI / NVIDIA / Alphabet / MIT Labs caliber. That's a multi-page site with bespoke hero composition, a content-management layer (Pages CMS) for Merric to swap hero media without code, a Spotify Now Playing widget (requires a Cloudflare Worker), a family-graph cross-link layer (per CLAUDE.md §10), per-page metadata (OG / Twitter / JSON-LD), favicon set, and accessibility / perf gates (Lighthouse 95+ on all four). Hand-rolling all of that as vanilla HTML/CSS/JS at this caliber is feasible but burns time on automation work that mature tooling solves out of the box.

2. **Operator authorization.** On 2026-04-30 the operator (Shane) explicitly loosened CLAUDE.md §3.2 / §4.2 / §4.3: "we should just make the best choices and future proof and all that good stuff... we don't need to be strict to the 2 sections for the tech decision." This ADR exercises that authorization.

The v1 doc's *spirit* — learn the web fundamentals, ship near-zero client JS, keep the source legible, don't introduce abstractions Merric can't read — must be preserved. The *letter* (no framework, no Tailwind, vanilla JS only) is what's being revisited.

## Decision

Adopt the following stack for merricstrough.com v2:

| Layer | Choice | Why |
|---|---|---|
| **Meta-framework** | **Astro 6** (per Shane decision; install latest stable at scaffold time, fall back to Astro 5.17 LTS only if Astro 6 stable has not yet released as of `npm create`) | HTML-first; zero client JS by default; multi-framework component islands when interactivity is genuinely needed; built-in image / sitemap / RSS / MDX / View Transitions / font integrations; **first-class Cloudflare Workers support in v6** (lines up with our Spotify Now Playing Worker); revamped dev server on Vite Environment API; stable Live Content Collections; static output deploys identically to vanilla on GitHub Pages |
| **Language** | TypeScript strict mode | Type safety with zero runtime cost in static builds; catches drift as the codebase grows; no ergonomics loss for a single-author project |
| **Styling** | Vanilla CSS with a design-token layer + CSS Modules per Astro component | Treats tokens as first-class; legible source; avoids Tailwind's class-name noise at this project's scale |
| **Content** | Astro content collections with Zod schemas | Markdown-first authoring with type-safe frontmatter; surfaces schema errors at build time; integrates with Pages CMS |
| **Lint + format** | Biome | Single tool, ~10x faster than ESLint+Prettier, 2026 default |
| **Tests** | Playwright (visual + a11y smoke via axe-core), Lighthouse CI | Cross-browser, automated in CI |
| **Admin / CMS** | Pages CMS (hosted) primary; github.dev power-user fallback; Sveltia self-hosted as Phase-4 graduation if needed | Zero infra, GitHub OAuth, mobile-friendly, free, replaceable |
| **Spotify Now Playing** | Cloudflare Worker with Spotify Web API + refresh-token flow | First piece of infra beyond GitHub Pages; free tier; Lee Robinson canonical pattern (or 2026 equivalent — confirmed by research subagent) |
| **CI/CD** | GitHub Actions: build → lint → typecheck → Playwright → Lighthouse CI → deploy to Pages | Standard, free for public repos |
| **Security automation** | Dependabot + CodeQL + OSV-Scanner; branch protection on `main` | Per CLAUDE.md §5.2 mandatory controls |
| **Analytics** | Cloudflare Web Analytics (free, privacy-first, no cookies, no GDPR concerns) | Optional; can defer to Phase 4 |
| **Hosting** | GitHub Pages (unchanged) | CNAME / HTTPS unchanged; Astro static output drops in |
| **Custom domain** | merricstrough.com via existing DNS (unchanged) | No DNS changes needed |

**Source-of-truth principle:** the repo contains Astro source (`src/`); the static build output (`dist/`) is generated by GitHub Actions on push and deployed via the official Astro deploy action. Source is canonical; `dist/` is ephemeral and gitignored.

## Alternatives considered

| Option | Why not |
|---|---|
| **Continue vanilla HTML/CSS/JS** | Defensible but burns hand-rolling time on image optimization, font subsetting, view transitions, content collections, sitemap generation — work Astro solves once. Loses the typed content authoring layer that makes Pages CMS sing. |
| **Next.js (App Router, static export)** | Fine for app-led sites; overkill for content-led personal site. Heavier build, more JS shipped by default (even with `'use client'` discipline), more abstractions Merric would have to learn before he can read the source. |
| **SvelteKit (adapter-static)** | Excellent framework; more app-oriented than content-oriented. Astro wins on content collections + integration ecosystem for this exact use case. SvelteKit components could still be used as Astro islands when needed. |
| **Eleventy (11ty)** | More minimal than Astro; great if we wanted maximum simplicity. Loses the component model and built-in image optimization that we'd otherwise hand-roll. Tighter call than the others — Eleventy is a respectable runner-up. |
| **Tailwind CSS** | Optimized for multi-author teams with 50+ components. Adds class-name noise at this scale; obscures the design-token discipline we want Merric to internalize. Vanilla CSS + tokens + CSS Modules wins on legibility for a single-author personal site. |
| **Astro DB + admin route + auth** | Moves us off GitHub Pages onto Cloudflare Pages or similar. Adds infra cost and complexity for capability we don't need yet. Pages CMS solves the admin problem with zero infra. |
| **Decap CMS / TinaCMS / Sanity** | Decap requires an OAuth proxy we'd have to host. Tina's free tier is more limited and visual editing is over-engineered for our schema. Sanity is overkill (real-time collaboration we don't need). Pages CMS is the right complexity match. |
| **Monorepo across family-domain properties (Nx, Turborepo)** | Premature. CLAUDE.md §3.3 calls for separate repos per subdomain / sibling property. Extract a `@strough/design` package only when 2-3 properties have shipped with duplicated tokens — not before. |

## Consequences

**Positive:**
- Image, font, sitemap, OG, MDX, View Transitions all become integrations rather than hand-rolled work
- Pages CMS gives Merric a clean admin UI day-one, fully GitHub-backed, no creds in browser
- Type-safe content collections catch authoring errors at build time
- Lighthouse 95+ becomes the default rather than the goal (Astro's perf defaults are strong)
- Stack is portable — if GitHub Pages ever stops serving us well, Astro static output deploys to Cloudflare Pages / Vercel / Netlify with zero code changes
- TypeScript catches drift as the codebase grows; reduces refactor risk

**Negative / costs:**
- Adds a build step. Source no longer renders in a browser without `npm run dev`. Mitigation: dev server is one command; preview is one command; Merric can author markdown content via Pages CMS without ever touching `npm`.
- Adds tooling Merric has to learn over time (Astro components, content collections, TypeScript). Mitigation: Astro's component syntax is essentially HTML-with-frontmatter. We document the on-ramp in `/docs/`.
- Cloudflare Worker for Spotify Now Playing is our first infra beyond GitHub Pages. Mitigation: free tier covers it forever; one-time setup; secrets in Worker env vars; documented in CLAUDE.local.md.
- Pages CMS hosted dependency. Mitigation: Sveltia self-hosted is the documented graduation path; takes ~2 hours to migrate if Pages CMS shuts down.
